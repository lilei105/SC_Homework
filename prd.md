# 财务报表问答系统 (Financial RAG) 产品需求文档

## 1. 产品概述

### 1.1 产品背景
在金融场景下，传统的 RAG 系统面临极高的挑战：财务报表包含密集的数值、复杂的表格以及跨页的逻辑论述；用户查询往往要求绝对的精确度（如“2024年第三季度某业务板块的营业利润”）。通用 RAG 架构（单一文本切分 + 稠密向量检索）在处理此类文档时，常出现上下文割裂、数值匹配失败和严重幻觉。

### 1.2 设计依据
本系统的核心架构设计深度参考了 FinSage [1] 与 VeritasFi [2] 两篇针对金融文档的顶会论文。为了适应快速迭代的工程需求，本方案对论文中的复杂模块进行了裁剪与优化，形成了一套“高精度、易部署”的本地与在线混合架构。

### 1.3 核心假设
系统输入已完成预处理：文档已通过 OCR 与版面分析工具（如 MinerU 或 PaddleOCR-VL）解析，并转化为按页结构化的 JSON 数据。每一页作为一个完整的、自然的语义单元（Chunk），平均长度在 200-400 tokens 之间。在此前提下，系统**无需进行带重叠（Overlap）的滑动窗口切分**。

## 2. 核心模块设计

### 2.1 索引管道（Indexing Pipeline）

索引管道的目标是将结构化的页面数据转化为支持多维度高精度检索的向量表示。

#### 2.1.1 块级元数据增强（Chunk-level Metadata Augmentation）
*   **需求描述**：孤立的财务数值缺乏上下文。必须为每个页面赋予宏观的语义标签，以提升检索的全局匹配度 [1] [2]。
*   **实现逻辑**：
    1.  调用大语言模型（如智谱 `glm-4.7-flash` API），输入单页的结构化文本。
    2.  模型输出标准化的标题摘要（Heading Summary），格式要求包含：公司名称、财报期间、本页核心主题（如 "Apple Inc. Q3 2024 - Revenue by Segment"）。
    3.  将生成的摘要作为元数据，直接拼接在页面原始文本的头部，形成增强后的待编码文本。

#### 2.1.2 BGE-M3 原生双路编码与存储
*   **需求描述**：金融查询既需要理解“语义”（如 Profit vs Income），又需要精确匹配“词汇”（如 EBITDA, $12.3B）。
*   **实现逻辑**：
    1.  加载本地部署的 `BAAI/bge-m3` 模型（通过 `FlagEmbedding` 库）。
    2.  对增强后的文本进行一次前向传播，获取稠密向量（Dense Vector）和稀疏权重（Sparse Lexical Weights）。
    3.  **存储至 Qdrant**：Qdrant 原生支持单条记录挂载多个命名向量（Named Vectors）。将 1024 维的稠密向量存入 `dense` 字段，将包含词汇权重的字典存入 `sparse` 字段。这种设计避免了维护两套独立索引的复杂性，且 Qdrant 的 Python 客户端（`qdrant-client`）提供了极简的本地单文件运行模式，非常适合快速原型开发。

*   **注：ColBERT 向量处理策略**
    BGE-M3 产出的第三种表征（ColBERT Multi-Vector）为变长张量，无法直接存入常规向量数据库。考虑到一份财务报表通常在 100-300 页之间，预先计算并存储所有页面的 ColBERT 向量不仅增加存储开销，且工程实现繁琐。因此，本方案**不持久化存储 ColBERT 向量**，而是将其推迟至重排序阶段，针对极少数候选页面进行**按需实时计算**。

### 2.2 检索管道（Retrieval Pipeline）

检索管道的目标是在海量页面中，精准、全面地召回与用户查询相关的候选页面，并恢复完整的财务逻辑链条。

#### 2.2.1 查询重写（Query Rewrite）
*   **需求描述**：用户的自然语言查询往往口语化且信息分散，直接检索效果不佳。
*   **实现逻辑**：调用大语言模型（`glm-4.7-flash`）对原始查询进行意图解析与重写，提取核心实体（公司、时间、指标），生成标准化的检索查询。

#### 2.2.2 原生混合初步召回（Native Hybrid Retrieval）
*   **需求描述**：结合语义与词汇的双重优势，初步筛选候选集。
*   **实现逻辑**：
    1.  使用 BGE-M3 对重写后的查询进行编码，获取查询的 Dense 向量和 Sparse 权重。
    2.  在 Qdrant 中执行并发混合检索（Hybrid Search）。
    3.  Dense 路召回 Top-50 页面；Sparse 路召回 Top-50 页面。
    4.  合并两路结果并去重，生成包含 50-80 个候选页面的集合。

#### 2.2.3 页面级块绑定（Page-level Chunk Bundling）
*   **需求描述**：解决跨页财务论述导致的信息碎片化问题，恢复文档的原始逻辑结构 [1] [2]。
*   **实现逻辑**：
    1.  遍历候选集中的页面 ID（页码）。
    2.  检测是否存在连续的页码（例如，同时召回了第 10 页和第 11 页）。
    3.  若存在，则在内存中将这些连续页面按顺序拼接，形成一个更大的、语义连贯的复合上下文块。

### 2.3 重排序与生成管道（Re-ranking & Generation Pipeline）

该管道的目标是对候选页面进行极限精排，并基于最相关的上下文生成最终的财务分析报告或精确答案。

#### 2.3.1 两级精细重排序
*   **需求描述**：初步召回的 50-80 个页面中仍包含大量噪声，需要通过深度语义交互进行过滤。
*   **实现逻辑**：
    1.  **一级重排序（ColBERT MaxSim 实时计算）**：由于候选集已缩小至 50-80 个页面，此时对这些页面重新调用一次 BGE-M3 提取 ColBERT 向量，并与查询的 ColBERT 向量计算 Token 级别的最大相似度（Late Interaction）。此步骤耗时在百毫秒级，可将候选集精准缩减至 Top-10。
    2.  **二级重排序（Cross-Encoder）**：将 Top-10 页面与查询拼接，输入本地部署的 `bge-reranker-v2-gemma` 模型进行深度交叉打分 [1] [2]。选取最终得分最高的 Top-3 页面作为最终上下文。

#### 2.3.2 答案生成（Answer Generation）
*   **需求描述**：基于精确的上下文，生成无幻觉的财务回答。
*   **实现逻辑**：
    1.  将 Top-3 页面（约 600-1200 tokens）作为上下文，输入智谱 `glm-4.7-flash` API。
    2.  **Prompt 约束**：
        *   强制模型仅基于提供的上下文回答问题，禁止使用内部知识。
        *   若回答涉及具体数值，必须在括号内标注来源页码。
        *   支持根据用户指令生成结构化的执行摘要（Executive Summary）或具体的指标对比。

## 3. 架构优势总结

本方案在 2-3 天的开发周期约束下，实现了学术深度与工程可行性的完美平衡：
1.  **摒弃无效切分**：确认了财务报表页面级的语义独立性，直接以页面为 Chunk，彻底避免了滑动窗口切分带来的表格破坏和逻辑截断。
2.  **优雅的多路召回**：通过引入 Qdrant，激活了 BGE-M3 模型的原生 Dense+Sparse 混合检索能力，避免了外挂传统 BM25 库带来的架构冗余与性能降级。
3.  **深度的上下文恢复**：忠实复刻了论文中的 Chunk Bundling 和基于 LLM 架构的交叉重排序技术，通过 ColBERT 的按需实时计算策略，在不增加存储负担的前提下确保了极高的排序精度。
